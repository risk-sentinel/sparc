# SPARC API Review and Automated Testing Procedure

The repeatable procedure used to keep SPARC's API documentation and automated test coverage aligned with code. Updated whenever the procedure itself changes; the running coverage state lives in [`INVENTORY.md`](INVENTORY.md).

This document satisfies the procedural deliverable in issue [#413](https://github.com/Rebel-Raiders/sparc/issues/413).

---

## When to run this procedure

- Before every minor release (`v1.x.0`) — guarantees the inventory ships in sync with code.
- After any PR that adds, removes, or changes the path/method of an `/api/v1/` route.
- When opening a ticket that claims an API gap — re-running the procedure either confirms the gap or shows it was already closed.

The procedure is intentionally cheap to re-run end-to-end (under five minutes for a developer with the repo checked out and Rails booted) so it can be part of routine PR review when API surface changes.

---

## Authoritative inputs

The procedure treats three artifacts as inputs and produces one output.

### Inputs

| Artifact | Authority | Where it lives |
|---|---|---|
| **Code routes** | Source of truth for "what API exists" | `bin/rails routes` (filtered to `/api/v1/`) |
| **Per-endpoint docs** | Source of truth for "what is documented" | `docs/api/endpoints/*.md` |
| **Postman collection** | Source of truth for "what is in the operator-facing collection" | `docs/api/sparc-api.postman_collection.json` |

### Output

| Artifact | Purpose |
|---|---|
| `docs/api/INVENTORY.md` | Cross-reference table; one row per logical endpoint, columns showing presence in each input. |

### Reproducible generation

The inventory body is generated by `bin/api_inventory_check.rb`, which reads the three inputs and emits the markdown table to stdout:

```bash
# Print the inventory table body
bin/api_inventory_check.rb

# Drift check — exits 1 if any row is undocumented or missing from Postman
bin/api_inventory_check.rb --check
```

The summary block at the top of `INVENTORY.md` is hand-maintained (it captures *why* the gaps exist, which the script cannot infer); the table body below the `Full inventory` heading is the script's output. When re-running, replace only the table body and update the summary numbers.

## Phase 1 — documentation review

### Step 1: extract code routes

Run `bin/rails routes` and filter to `/api/v1/`. Collapse `PATCH`/`PUT` pairs that route to the same `controller#action` (Rails registers both for `update` actions; they are aliases, not distinct endpoints).

Strip the `(.:format)` suffix Rails appends so paths match the documentation form.

### Step 2: extract per-endpoint doc coverage

For each route, look up the doc file matching its controller (e.g. `ssp_documents` → `ssp-documents.md`). Mark the route as documented when **any** of the following appears in the doc file:

- The exact route path (`/api/v1/ssp_documents/:id`)
- The slug-keyed form (`/api/v1/ssp_documents/:slug`) — some doc files use `:slug` for slug-keyed resources even though Rails routes use `:id`
- The action name (`show`, `update_fields`) on a word boundary

Controllers with no doc file at all are recorded as `NO (no doc file)` rather than `MISSING`, so the gap analysis distinguishes "no doc exists yet" from "doc exists but is incomplete."

### Step 3: extract Postman coverage

Walk the Postman collection JSON, recursively collecting every leaf `request`. Normalize each request's URL by replacing `{{var}}` placeholders with `:id` so it matches the routes table form. Mark each route as covered when its `METHOD /path` appears in the collected set.

### Step 4: produce the inventory

Combine the three signals into the table at the bottom of `INVENTORY.md`. Add a summary block at the top covering:

- Total endpoint count
- Per-input coverage percentages
- Per-controller-group breakdown of where the gaps are
- A "Phase 1 gap summary" section listing exactly which doc files / Postman folders need work to close the inventory

### Step 5: close the gaps

Each row marked `**MISSING**` becomes a documentation task. Group tasks by doc file (one PR per logical area: e.g. "back-matter-resources doc + collection update"). Reference issue #413 in every PR.

A doc update is complete when:

- Every code route appears in the doc file's "Endpoints" overview table
- Every code route has a per-endpoint subsection covering: path parameters, query parameters, request body schema (where applicable), response body schema, error responses, and `curl` example
- The doc file's authentication / authorization claims match the controller's `before_action` filters and Pundit policy

A Postman update is complete when:

- The collection has a folder per controller group
- Every code route has a request entry under the appropriate folder
- Each request includes a working example body for `POST`/`PUT`/`PATCH`
- The collection's `info.description` field reflects the current endpoint count

### Step 6: independent validation

Per the issue, an independent reviewer must spot-check at least 30% of the rows in `INVENTORY.md`. The validator picks rows at random, opens the controller / doc file / collection entry, and confirms the inventory's claim about each. Findings get filed as PR comments on the validation PR; resolution is a follow-up commit.

The validation is a non-trivial gate and should be run after the doc-update PRs land but before the procedure is declared "complete" for a given inventory pass.

## Phase 2 — automated testing

The Phase 2 work happens on branch `sparc-api-automated-testing-phase2` and produces a Python `pytest` suite that consumes `INVENTORY.md` as a coverage spine.

### Test framework

- **Language:** Python 3.12+
- **Dependency manager:** [`uv`](https://docs.astral.sh/uv/) (`uv.lock` committed; reproducible installs via `uv sync --frozen`)
- **Runner:** `pytest` 9+ via `uv run pytest`
- **HTTP client:** `httpx` (for both sync and async support)
- **Schema validation:** `pydantic` v2 against response schemas captured during Phase 1 doc work
- **Layout:** `tests/api/` at the repo root, one test module per controller group (`test_ssp_documents.py`, `test_back_matter_resources.py`, etc.)
- **Fixtures:** under `tests/api/fixtures/` — JSON request bodies, sample upload files, environment templates
- **Containerized runner:** `tests/api/Dockerfile` based on `ghcr.io/astral-sh/uv:python3.12-bookworm-slim` for fully isolated local + CI runs

### Coverage requirements per endpoint

For every row in `INVENTORY.md`, the Phase 2 suite must include at minimum:

| Test class | Asserts |
|---|---|
| Happy path | `2xx` status + response schema matches the doc-captured schema |
| Authentication | Request without token returns `401`; request with revoked/expired token returns `401` |
| Authorization | Request from a token whose role lacks the required permission returns `403` (or `404` for resource-not-visible cases) |
| Validation errors | Required-field-missing requests return `422` with the documented error envelope |
| Pagination / filtering | Where the index endpoint supports them, paged + filtered requests return the expected slice |
| Idempotency | Where applicable (e.g. `admin/refresh_credentials`), repeated identical requests return `unchanged` rather than re-mutating |

Tests are tagged so they can be run by phase (`pytest -m phase1` / `pytest -m phase2`) or by controller (`pytest -k back_matter_resources`).

### Running the suite

```bash
cd tests/api
uv sync --extra dev                    # one-time install
uv run pytest                          # full suite
uv run pytest -m phase1                # subset by phase
uv run pytest -m back_matter           # subset by controller marker
uv run pytest -k "test_admin_creates"  # subset by name pattern
uv run pytest -n auto                  # parallel via pytest-xdist
```

Containerized invocation (no local Python or uv install required):

```bash
cd tests/api
docker build -t sparc-api-tests .
docker run --rm --network=host --env-file .env sparc-api-tests pytest
```

See `tests/api/README.md` for the full reference.

### Test data and configuration

The suite is **completely self-contained**. Running it requires:

1. A SPARC instance reachable at a URL configured via `SPARC_TEST_BASE_URL`
2. An admin Bearer token configured via `SPARC_TEST_ADMIN_TOKEN`
3. A non-admin Bearer token configured via `SPARC_TEST_USER_TOKEN`
4. Optionally, `SPARC_TEST_RESET_DB=1` to recreate test data between runs

All fixtures (sample SSP Excel files, OSCAL JSON, OSCAL XML, KSI catalog seeds) ship in the repo under `tests/api/fixtures/`. No external downloads are required.

### Drift detection

`bin/api_inventory_check.rb` now treats the pytest suite as a third coverage signal. Re-running it on any commit produces a three-column inventory (docs / Postman / pytest); `--check` mode exits 1 if any row is missing in any signal.

```bash
bin/api_inventory_check.rb            # print full table to stdout
bin/api_inventory_check.rb --check    # exit 1 on any drift
```

This is the recommended pre-commit gate when changing any `/api/v1/` route — running the script catches a missing endpoint doc, an absent Postman entry, or an uncovered pytest module before the PR is opened.

### CI integration

The Phase 2 suite is designed to run in GitHub Actions against an ephemeral SPARC instance booted from the same Docker Compose stack used for development. A workflow that:

1. Boots the `web` + `db` + `redis` services
2. Seeds the test database
3. Generates admin and user Bearer tokens
4. Runs `uv run pytest` against the boot
5. Reports per-test results in the run summary, with a coverage delta against `INVENTORY.md`

is the natural next step. The workflow file itself is **not** part of Phase 2 — adding it requires explicit user approval per repo policy on CI changes — and is filed as a follow-up. Until that lands, `tests/api/Dockerfile` lets contributors run the full suite against any reachable SPARC instance with a single `docker run`.

Failures produce per-endpoint diffs (expected vs. actual response schema, status code, header values) so a failing PR can identify the regressing endpoint at a glance.

### Acceptance — Phase 2 done

- Every row in `INVENTORY.md` has at least one passing test in the suite
- Suite runs in under 10 minutes on the CI runner
- Suite is documented in `tests/api/README.md` with the exact command to run locally and the env vars it consumes
- A "covered by pytest" column is added to `INVENTORY.md` so future drift between code and tests is visible at a glance

## How the inventory and this procedure stay aligned

This document describes the *method*; `INVENTORY.md` records the *current state*. Both should be updated in the same PR whenever:

- A new controller or endpoint is added → run the procedure, regenerate `INVENTORY.md`, write the new doc / Postman / pytest entries in the same change
- The procedure itself evolves (new heuristic, new validation step) → update this file, then re-run with the new method and regenerate the inventory

Drift between the two — e.g. a procedural step described here that the inventory was not generated under — is itself a Phase 1 gap and should be filed as a follow-up.
